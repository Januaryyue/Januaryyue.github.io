# 综述

1. WEB_OMT通信丢包

现象描述

  • web端经常出现页面加载缓慢，甚至失败
  • web端index.cgi程序未响应
  • ARM这边显示数据发生timeout

原因分析

  • boa未将数据正确搬运至web端
  • boa处理数据等待过程中，占用CPU资源过多
  • CPU资源不够，会导致 Socket 收发包出现延迟、丢包或阻塞等问题

源码分析

while(1){
    ...
    if(!request_ready){
        ...
    }
    process_requests();
}
void process_requests(void){
    while(request_ready){
        ...
    }
}

由boa的源码可以看出，当web端没有数据传输的时候，boa程序其实相当于是在while(1)中进行空转，所以程序仅仅是在不断地占用 CPU，不释放 CPU，导致 CPU 占用率极高，系统负载升高，其他进程和操作系统难以得到足够的 CPU 时间，从而导致系统变得缓慢或不稳定，甚至会出现无响应的情况。

此外，while(1) 空转还会消耗系统的内存资源，因为每个进程都需要占用一定的内存空间，这些内存空间并没有被有意义地利用，导致系统资源的浪费。

解决方案

只要在设备空转的情况下有任务进行处理即可，比如加上是sleep函数，对CPU资源进行释放

while(1){
    ...
    if(!request_ready){
        ...
    }
    process_requests();
    Sleep(0,10);    //sleep 10us
}

结果验证

设备刷新页面不卡顿，index.cgi程序不再丢包，ARM也不再timeout，boa的CPU占用率得到极大缓解

2. ORAN设备bit文件加载失败

现象描述

ORAN设备在系统起来后加载FPGA.bit文件会导致设备卡死，导致设备更新FPGA只能通过烧写底层进行替换，风险极大，不好把控

原因分析

由于FPGA的器件一直在工作中，所以无法直接进行替换加载，会导致系统挂死

解决方案

在加载FPGA之前对FPGA进行复位，强行终止FPGA，再进行替换加载即可

3. 设备网卡出现丢包问题

问题现象

设备在运行过程中TX/RX会出现丢包，error值不断增加，会导致网卡丢包问题。上层应用增加的补丁是检测error值到一定范围即ifconfig down/up网卡进行重置。

原因分析

这是由于Mac层时钟和Data数据之间存在相位差，导致ARM核收包数据会出现解析错误，phy状态机将其标记为错误数据，并进行丢弃，导致设备网卡丢包

static int yt8511_config_init(struct phy_device *phydev)    //phy驱动初始化
{
	int ret;
	int val = 0;

	ytphy_write_ext(phydev, 0xc, 0x8057);       //配置rx delay

	ret = phy_write(phydev, 0x9, 0x0);

解决方案

根据上面源码分析，增加rx/tx delay和确定rx clk的选择，通过控制变量法进行双插线的绘制，求出交叉点，横纵坐标即为最合适的delay值

后续思路借鉴

在后续网卡设计中，对ping包丢包现象，可以采取该解决方案进行delay的确定，并在phy初始化时进行配置，原则上tx delay的影响较小，需在Mac端进行配置

4. ORAN设备tampstamp出现系统挂死

现象描述

ORAN设备在接入PTP源的过程中，极小概率会出现设备丢失tampstamp，然后设备系统挂死，重启才能恢复。

源码分析

int axienet_tx_hwtstamp(struct axienet_local *lp,
			 struct aximcdma_bd *cur_p)
{
	u32 sec = 0, nsec = 0, val;
	u64 time64;
	int err = 0;
	u32 count, len = lp->axienet_config->tx_ptplen;
	struct skb_shared_hwtstamps *shhwtstamps =
		skb_hwtstamps((struct sk_buff *)cur_p->ptp_tx_skb);

	val = axienet_txts_ior(lp, XAXIFIFO_TXTS_ISR);
	if (unlikely(!(val & XAXIFIFO_TXTS_INT_RC_MASK)))
		dev_info(lp->dev, "Did't get FIFO tx interrupt %d\n", val);		

	err = readl_poll_timeout_atomic(lp->tx_ts_regs + XAXIFIFO_TXTS_RLR, val,
					((val & XAXIFIFO_TXTS_RXFD_MASK) >=
					len), 0, 1000000);

	nsec = axienet_txts_ior(lp, XAXIFIFO_TXTS_RXFD);
	sec  = axienet_txts_ior(lp, XAXIFIFO_TXTS_RXFD);
	val = axienet_txts_ior(lp, XAXIFIFO_TXTS_RXFD);

由源码逻辑可以看出，设备在两种情况下进行读取RX FIFO寄存器的值

  • 设备在收到回PTP包请求时，尝试去获取tx interrupt，若没有收到中断信号，则打印后依旧访问FIFO值
  • 设备收到RX中断信号后，FIFO处于等待full package中，导致timeout

无论哪种状态出发，都不应该直接去读FIFO寄存器。因为FIFO寄存器未准备好时，是不允许直接访问读取数据的。

解决方案

以前的解决方案是在设备未收到中断情况下，函数直接返回-1，补丁如下：

	if (unlikely(!(val & XAXIFIFO_TXTS_INT_RC_MASK))){

		dev_info(lp->dev, "Did't get FIFO tx interrupt %d\n", val);	
		return -1;
	}

此种补丁未考虑到timeout情况，即FIFO寄存器未准备好情况下，但确实可以降低一部分复现概率。因此代码应该修正为：

int axienet_tx_hwtstamp(struct axienet_local *lp,
			 struct aximcdma_bd *cur_p)
{
	u32 sec = 0, nsec = 0, val;
	u64 time64;
	int err = 0;
	u32 count, len = lp->axienet_config->tx_ptplen;
	struct skb_shared_hwtstamps *shhwtstamps =
		skb_hwtstamps((struct sk_buff *)cur_p->ptp_tx_skb);

	val = axienet_txts_ior(lp, XAXIFIFO_TXTS_ISR);
	if (unlikely(!(val & XAXIFIFO_TXTS_INT_RC_MASK)))
		dev_info(lp->dev, "Did't get FIFO tx interrupt %d\n", val);		

	err = readl_poll_timeout_atomic(lp->tx_ts_regs + XAXIFIFO_TXTS_RLR, val,
					((val & XAXIFIFO_TXTS_RXFD_MASK) >=
					len), 0, 1000000);
	if (err) {
		dev_info(lp->dev, "%s: Didn't get the full timestamp packet",
			   __func__);
		goto skb_exit;
	}

	...
	
skb_exit:
	time64 = sec * NS_PER_SEC + nsec;
	memset(shhwtstamps, 0, sizeof(struct skb_shared_hwtstamps));
	shhwtstamps->hwtstamp = ns_to_ktime(time64);

修改逻辑为：当设备的FIFO寄存器未准备好的情况下(不分何种情况导致)，设备放弃该次纠偏操作，为packet包补充timestamp为0.

结果验证

后续ORAN设备以新底层为测试版本情况下，接入PTP源进行测试，两个月未复现该情况。修改前的概率为1周稳定复现一次。

5. ORAN设备PTP驱动及PTP控制逻辑设计

预研需求

原有1588时钟同步方案精度(±80ns)不满足现有项目需求，需要使用新的算法控制方案对时钟进行同步，精度要求在±20ns内

原有设计分析

原有设计方案逻辑为：FPGA的timer寄存器每1PPS周期里会给ARM这边发送时钟，然后由ARM这边通过PTP协议对timer进行时间校准
存在的问题：

  • 利用秒脉冲校准时间校准周期太长（最低校准周期1S）
  • 利用秒脉冲校准校准链条过长（1588->时间戳->秒脉冲->秒脉冲统计->频率调整）
  • PTP驱动严重依赖xilinx PTP模块驱动，不符合项目需求。

理论设计

  • 由PTP协议获取到数据传输delay和频偏offset
  • 使用PI算法对delay和offset进行控制优化，降低系统的波动和抖动，消除系统的稳态误差，减小控制器的过冲
  • 将delay和offset进行运算得出频偏freq通过PTP底层驱动写入FPGA timer寄存器中进行时间校准
  • 由于FPGA的timer是使用N850晶振当做输入clk,因此还需将freq通过滤波器处理后对N850进行调频，达到校准timer时钟源的目的

验证结果

使用新的PTP设计方案，M2/A3等ORAN设备的PTP精度能达到±10ns，满足预研项目预期需求，且rms值较为稳定。

6. PTP丢失源情况下进行精度保持

需求分析

ORAN设备在丢失PTP master源的基础上，由于没法进行实时校准，导致时间一长，会导致N850时钟漂移越来越大，timer的时钟频偏较大。

解决思路

使用N850中的holdover功能，在同步源失效时，时钟系统需要依靠自身的本地振荡器维持时钟频率和相位。

算法调试

  • 利用合适的算法对最近的freq进行统计，预判出合适的freq
  • 将该freq写入DPLL0和DPLL1的holdover控制寄存器中，地址分别为0x156、0x1d6
  • 系统丢失同步源之后，立马开启holdover功能，详情见0x124和0x1a4寄存器说明

验证结果

在断开设备外部同步时钟源的情况下，系统时钟(N850)的时钟偏移能保持稳定几分钟，然后逐渐扩大，但速度明显较之前缓和了许多。

7. PTP master跳变导致rms收敛太慢

现象分析

测试中发现，不论是之前设备还是新设计设备，均会出现当master设备自身时钟和GPS进行同步后，发生跨度特别大的跳变，设备的PTP同步master的时钟源会特别慢，一直以最大的调频幅度也无法进行同步(同步时间看master的跳变幅度)

解决思路

经过多次测试得出结论，是由于ORAN设备时钟源进行调频时，遇到master跳变幅度过大导致同步时间较长。因此，可以换种思路，当PTP收到同步包时发现时间跳变幅度过大时(这个offset值和领导商量过为100000)，直接将同步时间写入到FPGA timer寄存器中，直接新更新时间，然后dispatch，再进行port同步，然后PTP再进行同步

验证结果

当du时间进行人为设定跳变后，ORAN设备的PTP会立马识别出跳变，然后进行dispatch的同时修改系统时间，然后进行再次port同步，再次同步时间间隔不超过1s。 虽然这是由源master的异常导致设备PTP异常，但是我们需要做到实时同步，不受外界影响，强壮自身逻辑。

8. PTP状态上报及web端参数配置

需求分析

ORAN设备需要web端对PTP的配置参数、PTP模式等进行配置，并需要实时获取PTP状态和收敛情况(rms)

实现方式

使用共享内存机制，完成应用层对PTP控制程序的指令下达及参数反馈。

源码实现

struct ptp_config
{
   short int delay_asymmetry;
   char two_step_flag;
   char priority1; 
   char priority2;
   char clockAccuracy;
   char timeSource;
   char domain_number;
   char clock_classes;
   char ptp_profile;
   char  restart;
};

struct ptp_config * get_ppt_cfg_from_arm(void){

	int ptp_cfg_shm_id = -1;
	static struct ptp_config *ptp_cfg_shm_addr;

	ptp_cfg_shm_id = shmget(0x1243, sizeof(struct ptp_config), IPC_CREAT | 0666);
	if(-1 == ptp_cfg_shm_id)
	{
	    return NULL;
	}
	ptp_cfg_shm_addr = (struct ptp_config *)shmat(ptp_cfg_shm_id, NULL, 0);
	if(-1 == (int)ptp_cfg_shm_addr)
	{
		ptp_cfg_shm_id = -1;
		return NULL;
	}
	return ptp_cfg_shm_addr;

}

代码实现没啥好说的，无非就是将PTP中的数据与ARM中的数据进行交互。重点是PTP中的参数配置及模式调控逻辑(不细讲，略).

验证结果

web端刷新数据对比PTP状态打印完全一致，模式调整也已达预期。

12. 软件在线升级底层不生效问题(二次升级)

现象描述

ORAN设备web端升级底层程序时，往往会出现update.sh脚本似乎未执行，里面的升级逻辑未生效，升级失败，需要设备重启后再进行执行。

原因分析

由于设备在web端进行升级时，web中的cgi调用了太多了busybox程序，(往往是以system进行fork调用，这个过程需要进行上下文切换、内存分配、文件描述符等资源的管理，会产生额外的系统调用开销)，导致busybox中嵌套的函数被大量调用，可能导致系统负载过高，出现性能瓶颈，使得后面调用system会出现无资源可分配，表面上看是脚本未生效。

分析验证

在update.sh脚本中加入echo等简单命令，不太消耗CPU资源，结果显示，该脚本是已经被执行了的，只是烧写flash的命令无效化。此时，将web升级包中的其他包舍弃，只保留底层升级包，结果显示，是能正常进行升级底层的。可以部分侧面验证原因猜测。

解决方式

  • 尽量减少system命令对系统资源的浪费，多使用fork/exev等函数直接创建子进程来执行命令，减少系统消耗
  • 采用线程池等技术来控制并发数，避免同时执行大量的 busybox 命令，从而减少进程调度和内存开销。
  • 可以尝试使用更轻量级的工具替代 busybox 命令，例如 toybox 等，以减少系统的资源开销
    针对项目中的升级问题，由于我这边单属底层，只能使用创建现成话命令来满足需求，故我这边为升级flash编了一个应用程序。
int main(int argc, char *argv[]) 
{
	int file_len = 0;
	char *str1;

	file_len = strtol(argv[3], &str1, 16);
	nand_erase(argv[1],0,file_len);
	printf("mtd:erase ok!\n");
	nand_write(argv[1],argv[2],0);

验证结果

ORAN使用该程序放置update.sh脚本中进行升级，可以达到预期，不需要二次重启，大大减少升级时间，缩小升级风险。

14. E3等AM35x设备出现ARM没有打印问题

现状分析

HPRU/E3/KDDI_RU设备在系统起来后，ARM均无打印，所使用的debug_msg无效化

原因分析

于Ti am335x平台而言，其配置用户态tty是由busybox中的getty中进行配置，经查，该平台内核配置将busybox环境配置放在比较靠后的地方，（login前面），arm起得比较早，由linuxrc起来的一系列脚本直接拉起来，导致打印环境未配置好。

解决方案

int main()
{
    // 打开串口设备
    int fd = open("/dev/ttyS0", O_RDWR | O_NOCTTY);
    if (fd == -1) {
        perror("open");
        return 1;
    }

    // 配置串口参数
    struct termios options;
    tcgetattr(fd, &options);
    options.c_cflag = B9600 | CS8 | CLOCAL | CREAD;
    options.c_iflag = IGNPAR;
    options.c_oflag = 0;
    options.c_lflag = 0;
    tcsetattr(fd, TCSANOW, &options);

    // 重定向标准输入、输出和错误输出到串口
    dup2(fd, STDIN_FILENO);
    dup2(fd, STDOUT_FILENO);
    dup2(fd, STDERR_FILENO);

    // 使用标准输入输出进行通信
    printf("Hello, serial port!\n");
    char buf[128];
    scanf("%s", buf);
    printf("You typed: %s\n", buf);

    // 关闭串口设备
    close(fd);

    return 0;
}

ARM源码中增加串口重定向代码(参考以上本人测试代码)即可实现ARM程序打印需求

结果验证

通过E3/RU设备测试，ARM程序打印ok，符合预期要求，需求予以解决。

15. E3等AM335x设备光口不通问题

现象描述

E3设备启动时，cpri网口一直显示rx无数据，ping不通其他设备

原因分析

FPGA使用Jtag进行抓包，是可以抓到数据包，但是rx_en一直不稳定。经猜测可能是RMII2_CRS_DV没有拉高导致的。查看原理图，TS5A3157DBVR通过信号切换，进行
输入RMII2_CRS_DV，从而影响FPGA rx_en的稳定性。通过示波器进行测量，wait_ctrl脚在启动过程中有短暂拉高的情况。

解决方案

在内核启动之初，就直接将wait_ctrl引脚接管，拉低之后再起FPGA，此时可以确保TS5A3157DBVR切换到输入RMII2_CRS_DV脚，保证rx_en的稳定。

16. 修改并同步上层ecpri和cpri网口名称

需求描述

  • 由于上层代码缺少灵活性，比如将eth0当做cpri网口，直接是写死的。所以需要底层这边进行特定网口使用特定名称，保证上层代码的可复用性
  • ORAN这边对ecpri网卡需要做定制处理，名称为eth-oran

实现原理

  • 内核注册网卡的顺序是按照id号进行分配，从0开始
  • 注册的先后顺序是按照设备树中网卡设备的排列顺序来的，排在第一个有效网卡设备即为eth0,依此类推
  • 内核可以通过识别网卡设备，在注册设备时进行重命名，强行修改网卡名称

源码实现

有两种实现方式，分别是自定义网卡名称和修改默认网卡名称。由于领导未做要求，故选择了最简单的模式，直接修改名称即可。

17. M2RU设备DAS和ORAN平台统一底层设计

需求描述

  • 目前产线生产流程中，ORAN设备需要在DAS设备基础上进行烧写底层，才能转换成ORAN设备，增加了产线负担
  • M2RU设备需要做boot四备份系统，提高系统容错率
  • ORAN设备需要支持即使分区变化，仍然能够支持对外网设备实现web升级底层
  • 统一版本后，不用可以维护两个版本的底层，只需要下发一个即可

理论分析

  • 针对M2RU设备DAS和ORAN设备的FPGA程序完全不一样，只能采取DAS和ORAN设备树进行分开加载的策略
  • 采取环境变量形式对DAS和ORAN设备进行设备树辨析
  • 由于增加uboot备份，底层分区已改动，为了支持外网设备进行升级，kernel分区地址不予修改，支持按flash偏移进行升级
  • M2RU-ORAN设备的ecpri网卡，加载前需要加载好FPGA程序提供时钟，因此需要提前加载bit文件

底层实现

  • 在系统生成image时，在image头部插入两张设备树，一份是DAS的，一份是ORAN的，供uboot进行加载使用
  • uboot中增加在读取image设备树时，采用识别环境变量das_platform的形式来加载不通的设备树
  • ORAN设备还是DAS设备均可直接修改环境变量方式进行切换，或web端升级env分区，目前默认是ORAN设备
  • 若为ORAN设备，则在启动cpri网卡前加载bit文件，若上层有bit文件即加载上层bit文件，否则加载底层默认bit文件

具体源码实现，请参考相应的详细设计，此处只展示设计思路及可行性分析

结果验证

M2RU底层目前经过长时间稳定性测试，已基本完成测试，表明该底层设计方案是可行的。
